Pour profiter des fonctionnalités fournies par Auth0 Actions, vous pouvez devoir migrer votre code actuel depuis des Règles, des crochets ou des versions antérieures d’Actions. Cependant, vous devez être conscient des limitations actuelles lors de la migration entre les offres d’extensibilité héritées et les Actions.

For more information, see [Migrate to Actions](/docs/customize/actions/migrate).

## Différences spécifiques aux règles

* La réutilisation des fonctions entre les Actions n’est pas prise en charge actuellement, car chaque exécution d’Action est autonome. Vous devez définir explicitement les fonctions requises pour chaque Action.
* L’objet [`accessToken`](/docs/libraries/lock/lock-api-reference#getuserinfo-) est actuellement limité à 100 permissions.
* The event object in Actions does not currently provide an alternative for the following [context attributes](/docs/customize/rules/context-object)from within a Rule:

  + `context.connectionOptions`
* Actions only support the following properties within [`event.user.identities`](/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/post-login-event-object):

  + `connection`
  + `provider`
  + `user_id`
  + `profileData`
  + `isSocial`

<Callout icon="file-lines" color="#0EA5E9" iconType="regular">

If you are using additional properties within `user.identities` object in Rules, they are not supported within an Action. This includes any token from upstream IdP for federated users.

</Callout>

* Les demandes standard d’un `idToken` ou d’un `accessToken` ne peuvent pas être supprimées à l’aide d’une Action.
* Actions can only return `access denied`error codes and does not support returning an `unauthorized` error code.
* Les Actions permettent uniquement de modifier l’utilisateur principal entre les comptes liés.

## Hooks specific differences

* You can not modify scopes in a machine-to-machine client credential flowas was possible with a Hook.
