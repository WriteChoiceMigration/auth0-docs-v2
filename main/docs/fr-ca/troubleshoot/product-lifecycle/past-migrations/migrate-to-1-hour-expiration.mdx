Depuis le 21 février 2024, Auth0 impose une durée de vie maximale d’une heure aux flux de connexion de type redirection. Les flux de connexion qui prennent plus d’une heure expirent avec la connexion universelle et la connexion classique.

## Flux de connexion touchés

Actuellement, Auth0 permet à certains flux de connexion interactifs de durer jusqu’à trois jours :

* [Authorization Code Flow](/docs/get-started/authentication-and-authorization-flow/authorization-code-flow)
* [Authorization Code Flow with PKCE](/docs/get-started/authentication-and-authorization-flow/authorization-code-flow-with-pkce)
* [Cross-Origin Authentication](/docs/authenticate/login/cross-origin-authentication)
* [Device Authorization Flow](/docs/get-started/authentication-and-authorization-flow/device-authorization-flow)
* [Hybrid Flow](/docs/get-started/authentication-and-authorization-flow/hybrid-flow)
* [Implicit Flow with Form Post](/docs/get-started/authentication-and-authorization-flow/implicit-flow-with-form-post)
* [Security Assertion Markup Language (SAML)](/docs/authenticate/protocols/saml)
* [Web Services Federation Protocol (WS-Fed)](/docs/authenticate/protocols/ws-fed-protocol)

Starting 21 February 2024, Auth0 will enforce a maximum lifetime of 1 hour for the listed <Tooltip tip="Authorization Flow: Authorization grant (or workflow) specified in the OAuth 2.0 framework." cta="View Glossary" href="/docs/glossary?term=authorization+flows">authorization flows</Tooltip>. Transactions will expire after 1 hour. After expiration, operations from the end user's browser, such as input email and password or redirect back to [Auth0 Actions](/docs/customize/actions), etc., will:

* redirigées vers le chemin de connexion par défaut de l’application pour réinitialiser le flux en tant que nouvelle transaction de connexion
  Or
* Serve an error page if you have not configured the application's default login route or you are using Classic Login. The end user must return to your application to reinitiate the login transaction. You can customize this error page using <Tooltip tip="Universal Login: Your application redirects to Universal Login, hosted on Auth0's Authorization Server, to verify a user's identity." cta="View Glossary" href="/docs/glossary?term=Universal+Login">Universal Login</Tooltip>.

## Actions

When [redirecting users after logout](/docs/customize/actions/explore-triggers/signup-and-login-triggers/login-trigger/redirect-with-actions), redirect them to a page of your application that allows users to log in again based on some user action (e.g. click the Login button) instead of directly redirecting them to the Universal Login `/authorize` endpoint. The pattern of starting a new login transaction without an explicit user-initiated action is the main source of transactions taking too long to complete, as the new login attempt is left unattended in a browser window and can be subject to malicious attacks.

Si vous utilisez la connexion universelle :

* We recommend you [configure default login routes](/docs/authenticate/login/auth0-universal-login/configure-default-login-routes) for your applications in the <Tooltip tip="Auth0 Dashboard: Auth0's main product to configure your services." cta="View Glossary" href="/docs/glossary?term=Auth0+Dashboard">Auth0 Dashboard</Tooltip> or via the Auth0 <Tooltip tip="Management API: A product to allow customers to perform administrative tasks." cta="View Glossary" href="/docs/glossary?term=Management+API">Management API</Tooltip>. This method automatically starts login transactions and results in the best user experience as the login flow seamlessly restarts for active end users when the transaction has expired.
* If your configuration doesn’t allow for default login routes, we recommend you [customize Auth0 error pages](/docs/customize/login-pages/custom-error-pages) to provide a better user experience. You can use the error pages to instruct the user to return to your application to restart the login attempt.
* If you don’t do either of the above, we will display the [default Auth0 error page](/docs/authenticate/login/auth0-universal-login/error-pages):

Si vous utilisez la connexion classique, le gadget logiciel de connexion affichera une erreur. L’utilisateur devra alors retourner à l’application et recommencer la procédure pour effectuer la transaction, puisqu’elle est expirée.

## Examiner les journaux de locataires

If you want further insights on how many transactions and Applications are affected in your tenant, review the logs in your tenant under [Auth0 Dashboard > Monitoring > Logs](https://manage.auth0.com/#/monitoring/logs). Look for `Deprecation Notice` logs with this search: `type:depnote AND description:*Long*state*expirations*`. These results will point to login transactions and the applications (i.e. `client_id`) lasting longer than 1 hour.

Notez que ces journaux sont soumis à la limite anti-attaques et vous en verrez un par heure par `client_id`.

## Vérifier la migration

Après avoir configuré les chemins de connexion par défaut ou mis à jour les pages d’erreur Auth0 avec des instructions particulières pour vos utilisateurs finaux, vous pouvez activer ou désactiver l’expiration à tout moment avant le 21 février 2024 pour vérifier le comportement de connexion.

1. Navigate to [Auth0 Dashboard > Tenant Settings > Advanced](https://manage.auth0.com/#/tenant/advanced) and search for the Migrations section.
2. Toggle off **Long state expirations**. Disabling this setting limits transactions to last only 1 hour. Enabling this setting allows transactions to last up to 3 days.
